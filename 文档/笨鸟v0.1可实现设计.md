# 笨鸟 v0.1 可实现设计

本文档在 [笨鸟v0.1规范](d:/SB/文档/笨鸟v0.1规范.md) 的基础上，补充实现所需的固定细节。
目标不是扩展边界，而是把当前版本补到“可以直接开始编码”的程度。

## 1. 实现目标

笨鸟 v0.1 的最小实现目标是：

1. 接收一段受限短场景文本
2. 抽取对象、属性、关系、事件、场景提示五类初始子信号
3. 用稀疏空间激活和有限传播构造候选场景解释
4. 输出结构化解释结果与推理路径

v0.1 的实现不追求开放生成，只追求结构化语义还原。

## 2. 最小模块划分

建议最少拆成六个模块：

1. 文本预处理模块
   负责分句、清洗、标准化同义表达。
2. 子信号抽取模块
   负责从文本中提取五类初始信号。
3. 空间索引模块
   负责管理概念空间、属性空间、关系空间、场景空间及其边。
4. 推理模块
   负责激活、传播、竞争和剪枝。
5. 评分模块
   负责局部激活评分和全局假设评分。
6. 输出模块
   负责生成标准 JSON 结构和解释路径。

## 3. 输入处理流程

### 3.1 预处理

输入进入系统后先做四步预处理：

1. 分句
   使用 `。！？；` 以及必要时的 `，` 做浅层切分。
2. 标准化
   将“旁边”“边上”“边儿上”归并为“在……旁边”，将“洒出来”归并为“泄漏/洒落”。
3. 去噪
   去掉对 v0.1 无意义的语气词、重复助词和明显修饰性停用词。
4. 片段编号
   为每个句段生成固定的 `segment_id`，便于追踪信号来源。

### 3.2 信号抽取优先级

为了降低歧义，抽取顺序固定如下：

1. 对象词匹配
2. 属性词匹配
3. 关系模板匹配
4. 事件模板匹配
5. 场景提示词匹配
6. 未登录词挂接

对象优先于属性，关系优先于场景解释。
也就是说，先抽事实，再做解释。

## 4. 初始子信号抽取规则

### 4.1 初始子信号统一结构

所有初始子信号统一使用以下结构：

```json
{
  "signal_id": "sig_001",
  "signal_type": "object",
  "surface": "碎砖",
  "normalized": "碎砖",
  "category": "堆放物",
  "segment_id": "seg_1",
  "span": [5, 7],
  "confidence": 0.95,
  "anchors": [],
  "attributes": [],
  "provenance": ["text_parser"]
}
```

字段含义固定如下：

- `signal_id`：信号编号
- `signal_type`：信号类型，只能是 `object`、`attribute`、`relation`、`event`、`scene_hint`
- `surface`：原文片段
- `normalized`：标准化后的语义标签
- `category`：领域内语义类别
- `segment_id`：信号来自哪个句段
- `span`：原文字符区间
- `confidence`：抽取置信度
- `anchors`：与哪些对象或信号绑定
- `attributes`：补充槽位
- `provenance`：来源链

### 4.2 对象信号抽取

对象信号通过对象词表和别名字典抽取。

抽取规则：

- 精确命中对象词表：置信度 `0.95`
- 命中别名映射：置信度 `0.85`
- 命中未知名词模板但周围上下文强支持：置信度 `0.60`

未登录对象词允许先以临时对象信号进入系统。

### 4.3 属性信号抽取

属性信号包括颜色、形态、稳定性、整洁度、污染状态等。

抽取规则：

- 直接修饰对象时，锚定到最近对象
- 若无法直接锚定，则保留为浮动属性信号，等待关系或场景空间吸收

例如：

- “红色杯子” -> 属性 `红色` 锚定 `杯子`
- “地面裸露” -> 属性 `裸露` 锚定 `地面`
- “摆放杂乱” -> 属性 `散乱/不整齐` 锚定 `场景`

### 4.4 关系信号抽取

关系信号优先由模板生成。

固定模板示例：

- `X上有Y` -> `Y 在……上 X`
- `Y在X上` -> `Y 在……上 X`
- `X旁边有Y` -> `Y 在……旁边 X`
- `Y在X旁边` -> `Y 在……旁边 X`
- `X里有Y` -> `Y 在……里面 X`
- `靠近X` -> `当前对象 靠近 X`
- `X后面有Y` -> `Y 在……后方 X`

如果同一句中缺少显式主语，则优先回指最近对象。

### 4.5 事件信号抽取

事件信号也由模板生成。

固定模板示例：

- `X碰到了Y` -> 事件 `撞击`，`actor=X`，`target=Y`
- `X翻倒了` -> 事件 `翻倒`，`target=X`
- `X被碰倒` -> 事件 `翻倒`，`target=X`，并附事件前因 `撞击`
- `Y洒在Z上` -> 事件 `泄漏/散落`，`actor=Y`，`target=Z`
- `X堆着Y和Z` -> 事件 `堆放`，`location=X`，`targets=[Y,Z]`
- `Y散落在X上` -> 事件 `散落`，`target=Y`，`location=X`

### 4.6 场景提示信号抽取

场景提示信号只承担弱引导，不直接决定结论。

固定提示类包括：

- 位置提示：院子角落、屋后空地、路边、沟渠里
- 风险提示：居民房附近、门口、道路旁
- 状态提示：没有整齐堆放、看起来杂乱、周围有杂草

场景提示信号默认置信度低于对象和关系信号。

## 5. 未登录词处理

v0.1 中，未登录词只能走受控挂接：

1. 如果是名词形态，先生成临时对象信号
2. 如果与已有概念空间高相似，则映射到最近概念空间
3. 如果连续多次出现且无法稳定映射，则进入临时空间
4. 临时空间只有在多次复现后才允许升级

这里“不认识”不等于“创建新体系”。
v0.1 只允许小范围补词，不允许无边界扩张。

## 6. 空间数据结构

### 6.1 空间统一结构

```json
{
  "space_id": "concept_brick",
  "space_type": "concept",
  "label": "碎砖",
  "category": "堆放物",
  "aliases": ["砖块", "砖头碎块"],
  "required_signal_types": ["object"],
  "supporting_signals": ["破碎", "堆放", "建筑材料"],
  "rejecting_signals": ["液体", "食物"],
  "default_score": 0.60,
  "state": "stable",
  "out_edges": [
    {"target": "scene_construction_waste", "weight": 0.85, "relation": "supports"}
  ]
}
```

### 6.2 概念空间

概念空间负责回答“这是什么”。

必须包含：

- 名称
- 所属类别
- 别名
- 支持它成立的典型属性
- 与哪些场景空间存在强连接

### 6.3 属性空间

属性空间负责回答“它有什么特征”。

必须包含：

- 属性标签
- 可附着对象类别
- 冲突属性列表
- 对场景空间的支持权重

例如：

- `破碎` 支持 `建筑废弃物堆放`
- `翻倒` 支持 `倾倒/翻倒事件场景`
- `发黑` 支持 `污染/异常积水`

### 6.4 关系空间

关系空间负责回答“谁和谁之间怎样关联”。

必须包含：

- 关系类型
- 主体类型约束
- 客体类型约束
- 支持的场景解释
- 冲突关系列表

### 6.5 场景空间

场景空间是候选解释中心。

每个场景空间必须定义：

- 场景名称
- 典型支持对象
- 典型支持属性
- 典型支持关系
- 典型支持事件
- 明显排斥信号
- 基础先验分数

## 7. 边与传播规则

### 7.1 边类型

v0.1 只允许三种边：

- 支持边
- 抑制边
- 别名映射边

不允许更复杂的高阶边。

### 7.2 边权重范围

边权重固定在 `0.0` 到 `1.0` 之间。

推荐解释：

- `0.80 - 1.00`：强支持
- `0.60 - 0.79`：中强支持
- `0.40 - 0.59`：弱支持
- `0.20 - 0.39`：弱提示
- `< 0.20`：默认不传播

### 7.3 深度衰减

传播深度固定最多 4 轮。

每轮衰减系数固定为：

- 第 1 轮：`1.00`
- 第 2 轮：`0.80`
- 第 3 轮：`0.60`
- 第 4 轮：`0.45`

### 7.4 传播分数

子空间传播支持度定义为：

```text
传播支持度 = 父信号强度 * 边权重 * 深度衰减 * 上下文兼容度
```

若传播支持度低于 `0.30`，则不再继续传播。

## 8. 局部激活评分

局部激活评分固定定义为：

```text
局部激活分数 =
  0.35 * 类型匹配 +
  0.35 * 内容相似 +
  0.20 * 上下文兼容 +
  0.10 * 转移先验 -
  冲突惩罚
```

### 8.1 各项定义

- 类型匹配
  完全匹配记 `1.0`，可兼容记 `0.6`，不兼容记 `0.0`
- 内容相似
  精确命中 `1.0`，别名命中 `0.9`，同类近义 `0.7`，临时挂接 `0.5`
- 上下文兼容
  与当前对象、句段、位置关系一致性有关
- 转移先验
  来自上一步激活空间的支持，一般不超过 `0.3`
- 冲突惩罚
  与当前已接受状态冲突时取 `0.1 - 0.4`

### 8.2 激活阈值

局部激活分数大于等于 `0.55` 才允许激活空间。

每个初始信号只保留前 5 个得分最高的空间。

## 9. 全局假设评分

场景候选的总分固定为：

```text
全局假设分数 =
  0.22 * 对象支持度 +
  0.18 * 属性一致性 +
  0.20 * 关系一致性 +
  0.22 * 场景兼容性 +
  0.18 * 事件合理性 -
  0.10 * 冲突惩罚 -
  0.06 * 互斥惩罚 -
  0.04 * 路径复杂度惩罚
```

### 9.1 正向项定义

- 对象支持度
  目标场景所需对象被满足的程度
- 属性一致性
  场景支持属性和对象实际属性的匹配程度
- 关系一致性
  位置关系、包含关系、邻接关系是否成立
- 场景兼容性
  当前证据和场景原型是否整体相容
- 事件合理性
  事件链是否自洽，状态变化是否自然

### 9.2 惩罚项定义

- 冲突惩罚
  同一对象上出现相互矛盾状态
- 互斥惩罚
  同一候选中同时激活强互斥场景
- 路径复杂度惩罚
  依赖过多跳传播才成立的候选要轻度扣分

### 9.3 输出规则

- 始终保留 top-3 场景候选
- top-1 作为 `best_hypothesis`
- 若 top-1 与 top-2 分差小于 `0.08`，则在输出中标记“存在竞争解释”

## 10. 冲突与互斥规则

### 10.1 对象状态冲突

同一时刻默认禁止以下共存：

- `稳定` 与 `翻倒`
- `整齐` 与 `散乱`
- `干净` 与 `污染`
- `完整` 与 `破碎`

若文本明确包含事件链，则允许“前后状态变化”，但必须通过事件更新实现，而不是静态并存。

### 10.2 关系冲突

默认禁止以下冲突：

- 同一对象同时“在桌上”和“在沟渠里”
- 同一对象同时“在桶里”和“散落于地面”
- 同一对象同时“靠近建筑材料堆”和“位于室内餐桌中心”

### 10.3 场景互斥

v0.1 建议固定如下互斥对：

- `普通物体摆放` 与 `倾倒/翻倒事件场景`
- `普通物体摆放` 与 `建筑废弃物堆放`
- `污染/异常积水` 与 `普通整洁场景`

## 11. 事件引起的状态更新

为了处理简单场景变化，v0.1 固定一张最小状态更新表：

- `撞击(X, Y)`：提高 `Y` 的不稳定风险
- `翻倒(Y)`：将 `Y.稳定性 = 翻倒`
- `泄漏(水, 桌面)`：增加对象 `水` 与 `桌面` 的覆盖/散落关系
- `堆放(位置, 物体组)`：为每个物体增加 `堆积` 属性和位置关系
- `散落(物体, 位置)`：为物体增加 `散乱` 属性和 `散落于` 关系

状态更新顺序固定为：

1. 抽取事件
2. 更新对象状态
3. 再参与场景评分

这一步不能反过来。

## 12. 推理主流程伪代码

```text
输入文本
 -> 预处理
 -> 抽取五类初始子信号
 -> 为每个信号检索候选空间
 -> 计算局部激活分数并保留 top-k
 -> 执行最多 4 轮传播
 -> 构造场景候选
 -> 计算全局假设分数
 -> 保留前 10 个候选
 -> 输出前 3 个场景解释
 -> 生成最佳解释路径
```

## 14. 学习与参数更新细则

v0.1 只允许做轻量更新：

1. 新词映射
   新词连续出现至少 3 次，且 2 次以上被人工或规则映射到同一概念，才允许写入别名字典。
2. 边权微调
   每次更新幅度不超过 `0.05`。
3. 场景权重微调
   每次更新幅度不超过 `0.03`。
4. 临时空间升级
   只有当同一临时概念出现至少 5 次，且误匹配率低于 `0.20`，才允许升级为候选空间。

v0.1 不允许自动删除稳定空间。

## 15. 最小实现文件建议

如果开始写代码，建议直接对应以下文件：

- `sb/parser.py`
  文本预处理、词表匹配、模板抽取
- `sb/ontology.py`
  对象、属性、关系、场景空间定义
- `sb/scorer.py`
  局部激活评分和全局假设评分
- `sb/reasoner.py`
  稀疏传播、多假设竞争、路径保留
- `sb/state_update.py`
  事件导致的对象状态更新
- `sb/output.py`
  JSON 输出整理

## 16. 先实现什么

最推荐的顺序是：

1. 先把词表、模板和 JSON 输出跑通
2. 再实现局部激活和 top-k 空间筛选
3. 然后加入场景候选打分
4. 最后再补事件状态更新和受控学习

先得到稳定、可解释的错误，再考虑提高复杂度。

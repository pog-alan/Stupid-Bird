# 笨鸟 LLM 架构设计

本设计的目标，是把笨鸟从“结构化场景解释器”推进为“适用于 LLM 的记忆-检索-上下文编排架构”。

## 总体定位

笨鸟不直接替代大语言模型。
它更适合作为 LLM 的前置认知层，负责：

- 把输入拆成结构化语义结果
- 在长期记忆中检索相关概念与场景
- 把对话历史压缩成可控上下文
- 把这些内容编排成适合生成模型消费的消息包

因此，笨鸟在 LLM 体系中的角色是：

“结构化感知 + 长期记忆检索 + 对话上下文压缩 + 生成约束层”

## 分层架构

### 1. 输入理解层

由 `SBV01Engine` 完成：

- 对象识别
- 属性识别
- 关系抽取
- 事件抽取
- 场景假设竞争

这一层输出的是可审计的结构化结果，而不是直接生成自然语言。

### 2. 长期记忆层

由向量索引与知识库共同构成：

- 静态配置词表
- 后台学习升稳后的知识条目
- 向量化后的概念、属性、关系、事件、场景

这一层负责“可扩展、可检索、可长期积累”的语义记忆。

### 3. 对话状态层

由 `DialogStore` 维护：

- 当前会话轮次
- 最近几轮文本
- 最近主动提问
- 历史摘要

这一层不追求保存全部原始上下文，而是优先保留最近轮次和可压缩的摘要。

### 4. LLM 编排层

由 `SBLLMRuntime` 完成：

- 从结构化结果中提取事实
- 从检索结果中选取记忆块
- 从对话状态中生成历史摘要
- 组合 `system / developer / user` 三类消息
- 输出响应 schema

这一层是笨鸟真正“适用于 LLM”的关键。

## 关键设计原则

### 1. 不把所有上下文直接塞给模型

笨鸟应当采用：

- 有限工作上下文
- 长期外部记忆
- 检索式召回
- 摘要式压缩

而不是依赖无限扩张的原始对话窗口。

### 2. 结构化优先

LLM 消费的上下文不应只有自然语言描述，还应包含：

- 结构化事实
- 检索记忆
- 推理线索
- 不确定性
- 候选澄清问题

### 3. 回答必须可审计

生成层必须被限制在：

- 输入文本
- 结构化分析结果
- 检索记忆

之内。

如果证据不足，就输出不确定而不是补全幻想。

## 当前代码映射

- `sb/reasoner.py`：结构化解释与检索入口
- `sb/vector_memory.py`：长期记忆向量索引
- `sb/dialog.py`：会话状态与历史摘要
- `sb/llm_runtime.py`：LLM 上下文编排
- `sb/llm_bridge.py`：检索结果向 LLM 包装
- `sb/server.py`：HTTP 服务与 `/llm_context`

## 标准输出包

`/llm_context` 当前输出的重点包括：

- `messages`
- `response_schema`
- `memory_blocks`
- `metadata`

其中：

- `messages` 可直接送入 LLM
- `response_schema` 约束生成格式
- `memory_blocks` 便于调试和审计
- `metadata` 用于追踪生成模式和 grounding 规则

## 后续演进方向

1. 引入显式“事实缓存”和“会话摘要缓存”
2. 将工具调用计划也纳入 `developer` 消息
3. 增加输出验证层，校验回答是否超出 grounding 边界
4. 将笨鸟从单轮场景理解扩展到多轮任务型对话与复杂知识问答

# 笨鸟 SB-Core 课程训练初探论文草稿

## 摘要

为了推动 `SB-Core` 从短序列记忆原型走向真正的长上下文主干，一个直接想法是引入长度课程训练：先在短序列上训练，再逐步增加记忆跨度。本文记录了这一思路在 `SB-Core-Mini` 上的第一轮实验。我们将“固定长度训练”与“朴素课程训练”在相同总训练步数下进行比较，任务为 `passkey retrieval`。结果显示：在当前架构下，朴素课程训练并没有提升长上下文能力，反而明显破坏了训练长度内的收敛质量。这个负结果说明，`SB-Core` 的问题不只是训练长度不够，更关键的瓶颈在于写入策略、槽位保护和内容寻址机制。本文的价值不在于宣布课程训练无效，而在于明确指出：**课程训练只有和更强的记忆机制一起使用，才有可能真正帮助 SB-Core 跨长度泛化。**

**关键词**：笨鸟，SB-Core，课程训练，长上下文，负结果，记忆写入

## 1. 研究动机

在 [笨鸟SB-Core长上下文实验论文草稿.md](d:/SB/文档/笨鸟SB-Core长上下文实验论文草稿.md) 中，我们已经看到一个明确现象：

- `SB-Core-Mini` 可以在训练长度内学会 `passkey retrieval`
- 但在训练外更长序列上，`exact match` 仍为 `0.0`

一个自然推断是：也许模型只是“没有被教过更长长度”，因此可以通过课程训练逐步扩展记忆跨度。

这篇文档就是对这个推断的第一次实验性验证。

## 2. 实验问题

本文试图回答一个非常具体的问题：

**如果在总训练步数不变的前提下，把固定长度训练改为由短到长的课程训练，SB-Core 是否会获得更好的长上下文泛化能力？**

## 3. 实验设置

### 3.1 模型

实验使用的都是同一个 `SB-Core-Mini` 原型，配置与长上下文基准中的 `passkey` 实验保持一致：

- `d_model = 96`
- `state_dim = 96`
- `num_layers = 2`
- `semantic_memory_slots = 8`
- `working_memory_slots = 16`
- `router_top_k = 2`
- `dropout = 0.0`

实现文件：

- [core_lm_torch.py](d:/SB/sb/core_lm_torch.py)
- [train_lm.py](d:/SB/sb/train_lm.py)
- [v02_sb_core_curriculum_compare.py](d:/SB/examples/v02_sb_core_curriculum_compare.py)

### 3.2 任务

任务固定为 `passkey retrieval`。模型需要在早先位置看到一个 key，并在后面接收到查询标记后复现该 key。

### 3.3 训练策略

#### 固定长度基线

固定使用同一种训练长度，总共训练 `480` 步：

- `prefix_len = 6`
- `filler_len = 8`
- `key_length = 2`

#### 课程训练

按四个阶段逐步增加长度，总步数同样为 `480`：

1. `prefix_len = 4, filler_len = 4, steps = 120`
2. `prefix_len = 6, filler_len = 8, steps = 120`
3. `prefix_len = 10, filler_len = 14, steps = 120`
4. `prefix_len = 14, filler_len = 20, steps = 120`

### 3.4 评测集

评测仍采用三档长度：

- 训练分布：`prefix_len = 6, filler_len = 8`
- 长上下文：`prefix_len = 20, filler_len = 32`
- 更长上下文：`prefix_len = 28, filler_len = 48`

### 3.5 指标

仅对需要回忆的 key 输出段计算：

- `token accuracy`
- `exact match`

## 5. 结果解释

### 5.1 课程训练没有带来成功泛化

最直接的结论是：

- 课程训练并没有让 `SB-Core` 在长上下文上达到非零 `exact match`
- 它甚至没有保持住训练分布上的收敛

因此，“只靠逐步增加长度”并不能解决当前 `SB-Core` 的核心问题。

### 5.2 为什么 token acc 稍高却不是进步

表面上看，课程训练在长上下文上的 `token accuracy` 略高于固定长度基线。但这不意味着它真的更好，因为：

- `exact match` 仍然是 `0.0`
- 它在训练分布上的 `exact match` 几乎崩掉
- 更合理的解释是模型输出里偶然碰对了部分 token，而不是形成了稳定可检索记忆

所以这类提升更像噪声，不应被误判为成功。

### 5.3 真正缺的是记忆机制，不是训练顺序

这一轮负结果反而指向了更清楚的结构结论：

1. **写入覆盖问题没有解决**
   课程训练只改变了样本顺序，没有改变“工作记忆槽会被覆盖”的事实。

2. **内容寻址不足**
   当前模型主要依赖状态与槽位相似度做读取，但没有足够强的“key 绑定”机制。

3. **缺少写保护**
   重要记忆和短期噪声都会进入同一类槽位，课程训练越往后，越可能把前面形成的模式打散。

## 6. 结构启示

这次实验最有价值的地方，不是得到了一个好结果，而是排除了一个过于乐观的路径：

**SB-Core 不能把课程训练当成万能补丁。**

如果后续要继续推进，课程训练必须和以下改动一起出现：

- 内容寻址写入
- 槽位写保护
- 重要性门控
- episodic memory
- 路由辅助监督

也就是说，课程训练最多是“放大器”，不能替代记忆结构本身。

## 7. 下一步建议

基于这轮结果，推荐的下一步顺序是：

1. 把当前 `SBMemoryWriter` 从“按步轮转写入”升级成“按内容冲突与重要性选择写入”
2. 给工作记忆加入简单写保护，减少高价值槽位被覆盖
3. 在此基础上重新跑课程训练
4. 最后再和 tiny Transformer 重做公平对照

## 8. 结论

这轮课程训练初探得出的不是一个令人兴奋的正结果，而是一个更有方向感的负结果：

**对于当前 `SB-Core-Mini`，课程训练本身不足以带来长上下文泛化，真正需要优先修补的是记忆写入与保护机制。**

从研究推进的角度看，这并不是坏消息。它让我们少走了一条“只调训练顺序”的弯路，把注意力重新拉回到 `SB-Core` 最本质的地方：记忆是怎么被写入、保留和重新读出的。
